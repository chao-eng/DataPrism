# 任务开发文档：rou-tools

## 一、 全局开发阶段划分

本项目整体将分为 5 个核心迭代阶段进行：

* **Phase 1：工程基建与通讯联调 (1-2 Days)**
  * 初始化 Tauri + Vite + Vue 3 环境。
  * 打通前端到 Rust 后端的核心 IPC 通讯链路并搭建本地只读 SQLite 数据库环境。
* **Phase 2：工业极简 UI 规范实装 (2 Days)**
  * 根据 `项目架构设计文档` 对 CSS 变量（飞书色系）和全局 Typography 进行定义。
  * 产出：基于单纵列架构的核心页面 —— “任务看板”与“任务向导录入”界面。
* **Phase 3：核心数据提取引擎与算法基建 (Rust + JS Worker) (3 Days)**
  * 针对选中的大量 Excel 完成高并发大文件系统 IO 流水线。
  * 将 `process_excel.js` 上的“中位数 Z-Score 收束算法”翻译为高性能提取与落库后台服务。
* **Phase 4：大规模数据可视化渲染与强交互 (ECharts) (3-4 Days)**
  * 读取大容量 `data_points` 进行海量散点聚合下发。
  * 攻克 ECharts 散点拖拽范围选择与用户手动点击抹除数据库连带操作的交互难点。
* **Phase 5：任务状态合并闭环与导出引擎 (2 Days)**
  * 构建经过层层人工校准后的标准数据查询拼接并触发 `xlsx` 合并生成流程。

## 三、 核心难点解析与攻克方向

### 3.1 跨语言高密度通信卡顿点 (前端难点)
Tauri 中 Vue3 到 Rust (通过 invoke 通道) 在大段对象数组传递时非常耗费序列化运算性能。
**解法**：在 ECharts 初始化拿到十万级别图点时，应尽量少对单个对象装箱封箱。建议在 Rust/SQLite 层通过 group concat 组装两组并行长数组（X轴 Array 和 Y轴阵列传导），到达前端直接喂给 ECharts。

### 3.2 ECharts 散点点击精细度
用户想要在一个密集的散点星空中试图去掉唯一的那个极小误差刺点是异常痛苦的。
**解法**：开启 ECharts 的 DataZoom 滚轮缩放与平移游手，用户先进入出事地带缩小视野，利用 `tooltip` 的跟随机制确认此处的异常点然后再做精准剔除点击。

### 3.3 Node.JS 脚本转化架构挑战 (后端难点)
`process_excel.js` 原本是纯 Node 环境下基于内存强行运行的，甚至将庞大数组横向强转为 Excel 写入。基于 Tauri 后将不捆绑巨大的 Node.js 镜像引擎。
**解法方案**：
强烈建议**前端只负责展示壳，使用 Rust 重写原本的 Excel 解析器（如采用 `calamine` 或通过 WASM 跨语言）和那套中位数算法进行快速原生运算**。或者妥协化用前端的 Web Worker 将选定文件直接灌入纯 JS 进行算法过滤并借由 tauri 桥接写到本地 DB。为了稳定性，优先将计算职责移交至 Rust 后端处理队列。
